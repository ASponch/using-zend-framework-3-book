3.1. PHP Scripts

When you write a website in PHP, you put your code into a file with the .php extension. Such a file is called a PHP script.

For example, let's assume we want to implement a Login page for a simple website, and that we do not use a framework for that. Such a page would have a form with the E-mail and Password fields. Once the user submits the form, he passes the authentication and his identity is saved to PHP session. The script would look like below:

<?php 
// login.php
session_start();

// If user is logged in, redirect him to index.php
if (isset($_SESSION['identity']) {
    header('Location /index.php');
}

// Check if form is submitted.
$submitted = false;
if ($_SERVER['REQUEST_METHOD']=='POST') {
    
    $submitted = true;
    
    // Extract form data.
    $email = $_POST['email'];
    $password = $_POST['password];
    
    // Authenticate user.
    $authenticated = false;
    if ($email=='admin@example.com' && $password=='Secur1ty') {
        $authenticated = true;
        
        // Save identity to session.
        $_SESSION['identity'] = $email;
        
        // Redirect the user to index.php.
        header('Location /index.php');
    }
}
?>

<html>
    <head>
        <title>Login page</title>
    </head>
    <body>
        <?php if ($submitted && !$authenticated): ?>
            <div class="alert">
                Invalid credentials.
            </div>
        <?php endif; ?>
        <form name="login-form" action="/login.php" method="POST">
            <input type="text" name="email">
            <br>
            <input type="password" name="password">
        </form>
    </body>
</html>

The above scirpt is not only a typical example of a "pure" PHP script. It is also an example of how you should not write your websites. What's bad about it?

  1. This script tends to merge all the code into a single file. You do not have any separation of concerns. Intuitively, you understand that it would be more convenient to split the code responsible for user authentication and the code responsible for presentation (HTML rendering).

  2. This script has problems with security, because we didn't implement any validation of POST variables. PHP session is also subject to hacking. And the login.php PHP script will be located under the Apache document root directory, which is not very secure.

  3. This script doesn't use any PHP classes. Encapsulating functionality into classes in theory would make the code well structured and easy to support.

  4. In this script you have to write your own implementation of user authentication (and so on). Why do we reinvent the wheel and not use a well-designed library for that?

The above four problems are easily solved when you write a website within a framework (like Zend Framework 3).
