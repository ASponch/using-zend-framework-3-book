Как работает веб-сайт

3.1. Скрипты PHP

Когда вы пишете веб-сайт на PHP, вы помещаете свой код в файл с расширением .php. Такой файл также называется скриптом PHP.

Например давайте предположим, что нам нужно написать страницу Login для простого веб-сайта, и что мы для этого не пользуемся никаким фреймворком. Такая страница обычно имеет форму с полями E-mail и Password. Когда пользователь отправляет форму, он проходит процедуру аутентификации и его личность сохраняется в сессию PHP. Наш скрипт бы выглядел следующим образом:

<?php 
// login.php
session_start();

// Если пользователь прошел аутентификацию, перенаправить его index.php
if (isset($_SESSION['identity']) {
    header('Location /index.php');
}

// Проверить, не отправлена ли форма
$submitted = false;
if ($_SERVER['REQUEST_METHOD']=='POST') {
    
    $submitted = true;
    
    // Извлечь данные из формы
    $email = $_POST['email'];
    $password = $_POST['password];
    
    // Аутентификация пользователя
    $authenticated = false;
    if ($email=='admin@example.com' && $password=='Secur1ty') {
        $authenticated = true;
        
        // Сохранить личность пользователя в сессию
        $_SESSION['identity'] = $email;
        
        // Перенаправить пользователя на страницу index.php
        header('Location /index.php');
    }
}
?>

<html>
    <head>
        <title>Login page</title>
    </head>
    <body>
        <?php if ($submitted && !$authenticated): ?>
            <div class="alert">
                Invalid credentials.
            </div>
        <?php endif; ?>
        <form name="login-form" action="/login.php" method="POST">
            <input type="text" name="email">
            <br>
            <input type="password" name="password">
        </form>
    </body>
</html>

Вышеприведенный скрипт - это не только типичный пример кода на "чистом" PHP. Это пример того, как вам не нужно писать свои веб-сайты. Что в нем плохого?

  1. Этот скрипт плох тем, что в нем весь код слит в один единственный файл. Здесь нет никакого разделения ответственности. Интуитивно вы понимаете, что было бы удобнее разделить код, ответственный за аутентификацию пользователя и код, ответственный за презентацию (отображение HTML).

  2. Этот скрипт имеет проблемы с безопасностью, так как мы не реализовали никакую валидацию переменных POST. PHP сессия также может быть взломана. И кроме того, скрипт login.php будет помещаться в корневой директории документов Apache что не очень безопасно.

  3. Этот скрипт не использует классы PHP. Инкапсуляция кода в классы в теории сделала бы код более струкрутрированным и простым в поддержке.

  4. В данном скрипте вам приходится писать свою собственную реализацию аутентификации пользователя (и многого другого). Зачем изобретать колесо, и не воспользоваться хорошо написанной библиотекой для этого?

Вышеперечисленные проблемы легко решаются, когда вы используете фреймворк (такой как Zend Framework 3).
